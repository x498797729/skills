# PRD 输出格式规范

## 给人看的 PRD

### 核心原则
**让人一目了然，让团队能立刻行动。**

### 标题层级
- 一级标题（#）：大章节（一、背景与目标）
- 二级标题（##）：子章节（1.1 需求背景）
- 三级标题（###）：具体模块（大纲生成Agent）
- 不超过三级，避免层级过深

### 图表要求
- **必须包含**：设计流程图、系统处理流程图
- **推荐包含**：核心交互逻辑图、用户操作路径图
- **全部使用 Mermaid 代码块**，便于渲染
- 每个流程图上方加一句话说明图的目的

### 文字规范
- 每个功能点用一句话说清楚"做什么"
- 避免"可能"、"也许"、"大概"等模糊词
- 技术细节放备注或附录，正文只写业务逻辑
- 重点内容加粗
- 数据和指标必须具体（不写"提升效率"，写"工时从5天降至2天"）

### 篇幅控制参考
| 模块 | 建议篇幅 |
|------|---------|
| 基础信息 | ½ 页 |
| 背景与目标 | 1~2 页 |
| 用户画像 | 1 页 |
| 功能规划 | ½~1 页 |
| 流程设计 | 1~2 页（以图为主） |
| 详细功能需求 | 3~5 页（核心内容） |
| Agent/提示词设计 | 2~4 页（按Agent数量） |
| 数据需求 | 1 页 |
| 非功能需求 | ½ 页 |
| 兜底策略 | ½~1 页 |
| 埋点设计 | 1 页 |
| 角色权限 | ½ 页 |

### 表格使用规则
- 对比类信息用表格（如功能清单、权限矩阵、兜底策略）
- 叙述类信息用段落（如需求背景、用户画像）
- 表格列数不超过6列，保持可读性

## 给 AI 看的 PRD

### 核心原则
**结构化、无歧义、机器可解析。**

### 格式强制要求
- 所有标题用 `#` 标记
- 列表统一用 `-`
- 代码和数据结构用代码块
- 所有流程图使用 Mermaid 代码块

### 数据结构定义
用 JSON 格式明确输入输出：
```json
{
  "input": {
    "user_query": "string - 用户输入的自然语言描述",
    "context": "array - 关联的参考文档ID列表",
    "user_id": "string - 当前用户ID"
  },
  "output": {
    "result": "string - 生成的内容",
    "confidence": "float(0-1) - 置信度",
    "source": "array - 引用来源列表",
    "fallback_triggered": "boolean - 是否触发兜底"
  }
}
```

### 枚举值明确化
不写"根据情况返回不同状态"，必须列出所有可能值：
```
状态枚举：
- success: 正常返回结果
- low_confidence: 置信度低于阈值，触发人工审核
- timeout: 超时，触发降级方案
- error: 系统异常，返回兜底文案
- partial: 部分结果可用，标注缺失部分
```

### 提示词格式
```markdown
## Prompt: [Agent名称]

**Role**：[角色定义]
**Inputs**：
- {{变量1}}: [类型] - [说明]
- {{变量2}}: [类型] - [说明]

**Task**：[任务描述]

**Output Format**：
{
  "field1": "[类型] - [说明]",
  "field2": "[类型] - [说明]"
}

**Constraints**：
- [约束1]
- [约束2]

**Fallback**：[异常处理逻辑]
```

### 接口定义格式
```markdown
### API: [接口名称]
- **Method**: POST
- **Endpoint**: /api/v1/[path]
- **Request Body**:
  ```json
  {
    "field": "type - description"
  }
  ```
- **Response**:
  ```json
  {
    "status": "string - success|error",
    "data": {}
  }
  ```
- **Error Codes**: 400, 401, 500
```
