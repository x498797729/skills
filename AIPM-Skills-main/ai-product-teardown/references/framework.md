# 六层拆解框架详细说明

## 框架总览

| 层级 | 动作 | 产出物 | 解决的核心问题 |
|------|------|--------|-------------|
| 市场层 | 看市场研报 | 立项决策依据、竞品分析报告 | 赛道是否值得进入？竞争对手有谁？ |
| 商业层 | 看官网介绍+创始人访谈 | 一句话定位、商业模式 | 这是什么生意/机会？如何变现？ |
| 用户层 | 按场景深度体验 | 用户旅程图、评测打分表 | 用户真正完成的任务是什么？替代了什么？ |
| 技术层 | 拆技术流程+Agent组成 | 技术流程图、Agent拆解、上下文Schema | 怎么实现的？架构是什么？ |
| 模型层 | 分析调用的模型 | 模型路由表、选择理由 | 哪些环节依赖模型？如何节省成本？ |
| 基础层 | 分析调用的数据 | 数据资产清单、飞轮分析 | 数据飞轮是否存在？护城河在哪？ |

---

## Layer 1：市场层 — 赛道值不值得进入？

### Claude搜索任务

依次搜索以下关键词（根据产品调整）：

```
搜索1: "{产品名} 行业报告" 或 "{赛道} 市场规模 2024 2025"
搜索2: "{产品名} 融资" 或 "{产品名} 投资"
搜索3: "{赛道} 竞品" 或 "{赛道} 竞争格局"
搜索4: "{赛道} 技术成熟度" 或 "{赛道} Gartner Hype Cycle"
```

### 整理为结构化信息

- **市场规模**：当前规模、增长率、预测
- **竞争格局**：主要玩家、各自定位、融资情况
- **技术成熟度**：该赛道处于什么阶段（概念期/膨胀期/低谷期/爬升期/成熟期）
- **关键趋势**：影响赛道的核心变量

### 引导用户输出

> 基于以上信息，你觉得这个赛道目前处于什么阶段？值得进入吗？主要的竞争对手有哪些？如果你来做这个赛道，你的切入点会是什么？

### 简化规则

- 如果拆解目的是**面试准备**，市场层只需一段话概述，不必深入
- 如果搜索不到市场数据，记录为「信息缺口」，提示用户查看 `info-sources.md` 中的渠道自行获取

## Layer 2：商业层 — 这是一个什么样的生意？

### Claude搜索任务

```
搜索1: "{产品名} 官网" → web_fetch 官网首页和定价页
搜索2: "{产品名} 创始人 访谈" 或 "{创始人名} 采访"
搜索3: "{产品名} 产品发布 报道"
搜索4: "{产品名} 定价 收费" 或 "{产品名} pricing"
```

### 提取关键信息

1. **一句话定位**：为[谁]在[什么场景]下解决[什么问题]？
2. **商业模式**：变现方式、定价策略、成本结构
3. **护城河分析**：技术壁垒？数据壁垒？网络效应？品牌？工艺知识？
4. **产品气质**：创始人访谈中透露的价值观、审美取向、战略方向

### 引导用户输出

> 现在请你用一句话描述这个产品的定位（谁+场景+痛点）。然后说说你觉得它的商业模式是什么？它和市场上其他产品的核心差异点在哪里？它凭什么能活下来？

### 追问方向

- 如果用户说"它就是个套壳的"，追问："那它的壳（Agent编排、提示词工程、交互设计）本身有没有价值？如果模型达到100分，这个壳的作用是什么？"
- 如果用户对定价不确定，引导："如果让你来定价，你会怎么定？为什么？"

## Layer 3：用户层 — 用户真正完成的任务是什么？

### 3.1 生成评测维度（前置步骤）

在用户体验之前，先协商评测框架。详见 `eval-dimensions.md`。

### 3.2 设计体验任务清单

根据拆解范围，为用户设计具体测试case：

```markdown
## 体验任务清单

### 场景1：{核心场景名称}
- **目标**：测试{具体能力}
- **输入**：{具体的测试内容/prompt}
- **关注点**：
  - 流程中经过了哪些步骤？记录每一步的界面变化
  - 哪些地方需要你做选择/确认？（人机协作节点）
  - 有没有看到"更新字段信息"或类似的状态提示？
  - 哪些地方你觉得不满意？具体是什么问题？
  - 最终输出结果的质量如何？
```

### 3.3 用户回来后的引导

**结构化流程梳理**：
1. 先让用户口述完整流程
2. Claude帮助整理为**正向流程图**
3. 追问逆向流程：哪些节点可以回退？回退后系统怎么处理？
4. 整理为**逆向流程图**

**追问细节的关键方向**：
- "你说它给你推荐了N个选项，它是按什么逻辑推荐的？你试过输入不同内容看推荐是否变化吗？"
- "它在生成过程中有没有透出思考过程？比如'展示思考过程'或'更新字段信息'？"
- "不同步骤之间切换时，之前的信息有没有被保留？还是丢失了？"
- "你有没有试过故意给一个模糊/不完整的输入，看它怎么处理？"

**挑战判断**：
- "你给它打了X分，扣掉的分是因为什么？如果你是PM，这个问题你会怎么解决？"

### 3.4 不愿体验的降级方案

如果用户明确不愿体验：
1. 搜索 "{产品名} 评测" "{产品名} 体验" "{产品名} review"
2. 搜索应用商店评论、小红书/知乎用户评价
3. **必须在文件开头标注**：

> ⚠️ 本层分析基于公开评测文章和用户评价，非亲身体验。结论的置信度有限，建议后续亲自验证。

## Layer 4：技术层 — 它背后是怎么实现的？

这是最核心的拆解层。以下是完整的推导路径。

### 4.1 判断架构类型

提供四象限框架让用户判断：

| | Context确定性高 | Context确定性低 |
|---|---|---|
| **Workflow确定性高** | 固定工作流/RPA | 工作流+AI节点（如智能客服） |
| **Workflow确定性低** | 单Agent自主规划（如Deep Research） | Multi-Agent复杂协作（如OiiOii） |

引导问题：
> 基于你在用户层的体验，你觉得这个产品背后是固定Workflow、单Agent、还是Multi-Agent？你的判断依据是什么？

追问验证：
- "每次你给同样的输入，它的处理流程是否完全一样？如果一样，可能是固定Workflow"
- "它有没有展示出'思考'或'规划'的过程？如果有，可能是Agent"
- "你有没有看到多个不同'角色'或'身份'在工作？如果有，可能是Multi-Agent"

### 4.2 拆解Agent组成

引导用户：
> 在你的使用过程中，你能分辨出有几个不同的'角色'在工作？每个角色分别负责什么？想想：如果这是一个人类团队，需要哪些岗位？产品是如何将这些岗位映射为Agent的？

用户输出后，Claude帮助结构化：
- 列出所有Agent及其职责（使用 `templates/agent_card.md`）
- 梳理Agent之间的调用关系：链式？主从？并行？
- 标注Human-in-the-Loop节点

### 4.3 逐个拆解核心Agent

对每个关键Agent，引导用户依次分析：

**第一步：输入输出**
> 这个Agent的输入是什么？它的输出是什么？

**第二步：处理逻辑**
> 从输入到输出，它中间做了哪些关键步骤？

**第三步：工具集**
> 它调用了哪些工具？你是从哪些线索推断出来的？

**第四步：规则和边界**
> 它有什么不能做的？有什么必须遵守的规则？

**第五步：范式判断**
> 这个Agent更像Plan-and-Execute（先列计划再执行）还是ReAct（边想边干）？

Plan-and-Execute特征：展示Checklist逐步打勾、明确状态管理、支持断点续传、适合长链路高耦合任务

ReAct特征：大量中间推理、可能自我修正、每步都可能改变方向、适合探索性开放式任务

### 4.4 推导全局上下文（Context Schema）

**在所有Agent拆解完成之后进行。**

> 现在我们拆完了所有Agent，回头想想：这些Agent之间共享了哪些信息？哪些数据需要统一管理？

推导方法：
1. 从每个Agent的输入中提取需要读取的变量
2. 从每个Agent的输出中提取会写入/更新的变量
3. 找交集：被多个Agent引用的变量 → 全局上下文候选字段
4. 分类整理为Schema（使用 `templates/context_schema.md`）

> 注意：虽然最后才推导出Schema，但在最终文档中放在技术层最前面——因为理解了全局上下文，后面每个Agent的逻辑更容易理解。

### 4.5 推导提示词（可选，仅限复刻/深度学习目的）

> 你先写一版骨架——列出角色定义、关键步骤、工具调用规范、交互规则。

**核心理念：提示词不是写出来的，是从流程推导出来的。骨架比文笔重要得多。**

### 4.6 发现产品问题

问题定位框架：提示词问题？工具调用问题？流程设计问题？模型能力问题？

## Layer 5：模型层 — 哪些环节依赖什么模型？

### Claude搜索任务
```
搜索1: "{产品名} 模型 底层" 或 "{产品名} 技术栈"
搜索2: "{产品名} API" 或 "{产品名} model"
搜索3: 产品UI中出现的模型缩写
```

### 引导用户输出
> 有没有看到模型名称提示或缩写？生成不同内容时速度和风格有没有差异？

### 关键讨论问题
1. "哪些环节**必须**依赖强模型？哪些只是流程设计聪明？"
2. "如果你来做模型选型，评测维度是什么？"
3. "它是否根据任务复杂度动态切换模型？"
4. "如果模型全部达到100分，这个产品的价值还剩什么？"

---

## Layer 6：基础层 — 数据飞轮是否存在？

### 引导用户输出
> 这个产品要运转起来，需要哪些数据支撑？数据从哪来？用户越多数据优势越强吗？

### 整理框架
- **知识库**：风格库/模板库、专业知识库、模型表现知识
- **数据资产**：用户数据、内容数据、训练数据
- **数据飞轮**：用户使用→产生数据→改善产品→吸引更多用户→飞轮是否成立？

### 关键洞察
> 全局上下文的一致性协议在AI产品中尤其重要。视觉与情绪的映射、风格库的动态维护、黄金case的积累——你觉得这个产品做得怎么样？
