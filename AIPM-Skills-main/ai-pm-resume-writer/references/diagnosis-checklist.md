# 简历诊断与优化检查清单

## 诊断流程

收到用户简历后，按以下顺序扫描，标记所有命中的问题，然后按严重程度排序反馈。

---

## 一、致命问题（必须修复）

### 1.1 流水账式职责罗列
**症状：** 通篇"负责XX"、"参与XX"开头，无成果、无数据
**诊断标准：** 工作/项目经历中，超过50%的bullet point没有量化结果
**修复方向：** 每条改为 [动词] + [工作内容] + [方法] + [量化成果]

❌ 负责产品需求文档的撰写，参与产品迭代与优化
✅ 独立撰写「智能QA助手」模块PRD（3个功能点），文档被评价为「逻辑清晰」，直接进入开发

### 1.2 个人优势空洞无物
**症状：** 写"认真负责"、"勤奋努力"、"学习能力强"、"上进心强"等
**诊断标准：** 个人优势部分没有任何具体数字或具体项目名称
**修复方向：** 每句优势 = [经验/能力] + [关键动作] + [亮眼成果]

### 1.3 角色过度包装
**症状：** 实际只负责一个模块但写成"主导整个项目从0到1"
**诊断标准：** 面试时无法回答该项目的全局性问题
**修复方向：** 诚实写明"在XX项目中，我主要负责XX部分"

### 1.4 数据造假或经不起追问
**症状：** 数据过于夸张（如"提升300%"但无法解释计算方式）
**修复方向：** 宁可保守但真实，每个数据都要能解释"怎么算出来的"

---

## 二、严重问题（强烈建议修复）

### 2.1 缺乏AI特色
**症状：** 简历内容与传统PM无区别，看不出AI产品经理的独特能力
**检查项：**
- [ ] 是否提到过Prompt设计/优化？
- [ ] 是否有模型评估/选型的经历？
- [ ] 是否有数据标注规则设计？
- [ ] 是否有AI效果验证（A/B测试、评测集）的经历？
- [ ] 是否有RAG/Agent/知识库等AI产品设计？
**修复方向：** Action中加入AI PM独特工作内容

### 2.2 只写工具不写思维
**症状：** "熟练使用ChatGPT、Midjourney、Stable Diffusion等AI工具"
**修复方向：** 写你如何用AI解决问题，而不是你会哪些AI工具

❌ 熟练使用ChatGPT、Midjourney等AI工具
✅ 设计AI回复质量评估体系（事实准确性/专业术语正确性/回复完整度三维度），标注300+条评测数据

### 2.3 个人优势与工作/项目经历脱节
**症状：** 个人优势写了"数据驱动"，但工作/项目中没有任何数据分析的体现
**诊断方法：** 逐条检查个人优势中的关键词，看是否在下方经历中有对应支撑
**修复方向：** 个人优势是总括，经历是分论点，必须互相印证

### 2.4 Side Project写得太随意
**症状：** "用Coze搭建了一个小红书文案生成Bot"一笔带过
**修复方向：** 用完整STARL框架写——谁的问题、怎么做的、效果如何

### 2.5 未针对JD适配
**症状：** 一份简历投所有岗位，没有根据JD调整关键词和亮点排序
**修复方向：** 每投一个新职位，都根据JD微调关键词和经历排序

---

## 三、体验问题（建议优化）

### 3.1 篇幅失控
**标准：**
- 1-3年经验：建议1页
- 3-5年经验：建议1-2页
- 超过2页：必须精简

### 3.2 排版不专业
- [ ] 字体是否统一？
- [ ] 页边距是否足够？
- [ ] 重点信息是否加粗？
- [ ] 是否使用PDF格式？

### 3.3 信息冗余
- [ ] 是否写了政治面貌/婚姻状况/身高体重？（应删除）
- [ ] 邮箱是否专业？（避免随意昵称）
- [ ] 是否有与目标岗位完全无关的经历？（应删除或缩减）

### 3.4 叙事不连贯
- [ ] 个人优势→工作经历→项目经历是否讲述一个统一的故事？
- [ ] 是否有明确的"人设"？
- [ ] 各段经历之间是否有成长逻辑？

---

## 四、JD适配检查

当用户提供了目标JD时，进行以下匹配检查：

1. **关键词匹配度**：JD中的核心关键词在简历中出现了几个？
2. **技能匹配度**：JD要求的硬性技能，简历中有几个能找到印证？
3. **经验匹配度**：JD要求的经验年限和行业经验是否匹配？
4. **亮点排序**：简历中最突出的经历是否恰好回应了JD的核心需求？
5. **缺口识别**：JD要求但简历中完全没有体现的能力有哪些？如何补充？

**输出格式：**
```
匹配度评估：XX%（高/中/低）
✅ 已匹配：[列出已匹配的关键词和技能]
⚠️ 部分匹配：[有相关经验但表述不够突出的]
❌ 缺失：[JD要求但简历完全没有体现的]
建议调整：[具体的修改建议]
```

## 五、分层适配检查

根据用户的经验层级，检查是否用了正确的表达方式：

### 15-20k常见错误
- 用了"主导"、"规划"等超出实际角色的词
- 没有写清楚自己在团队项目中的具体贡献边界
- Side Project一笔带过，没有STARL结构

### 20-30k常见错误
- 仍然停留在"做了什么"，没有展示"为什么这么做"的思考过程
- 缺少完整的数据链路（问题→动作→数据结果）
- 技术理解部分写得太浅，与初级无区别
- **缺乏指标思维**：bullet只是动作罗列（做了A、做了B、做了C），没有呈现"发现指标问题→分析→行动→指标提升"的链路
- **数据孤立**：写了结果数据但看不出它服务于哪个上层业务目标

### 30-40k常见错误
- 视角仍停在功能层面，没有上升到业务/商业价值
- 缺少团队赋能、体系搭建方面的描述
- 没有展示方法论和可复用的能力
- **缺乏指标树结构**：各条bullet之间没有指标层级关系，看不出子指标如何服务顶级指标
- **缺乏体系化表达**：只写了零散动作，没有展示"我搭建了什么体系/机制来持续驱动指标"
- **不值得追问**：面试官看完觉得"就这样了"，而不是"每一条都想问问细节"——说明缺少决策逻辑和取舍过程
